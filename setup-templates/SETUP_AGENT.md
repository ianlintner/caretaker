# Caretaker Setup Guide

This guide is intended to be used by `@copilot` when setting up caretaker in a consumer
repository. Read this document fully before taking any action.

---

## What you are doing

You are installing the **caretaker** autonomous repository maintenance system into this
repository. Caretaker runs as a GitHub Actions workflow and uses Copilot to manage PRs,
triage issues, handle upgrades, and keep the repo healthy — all without human intervention.

---

## Step 1 — Determine the latest version

Fetch the release manifest from:

```
https://raw.githubusercontent.com/ianlintner/caretaker/main/releases.json
```

Use the highest `version` value from the `releases` array. This is the version you will
pin in `.github/maintainer/.version`.

---

## Step 2 — Create the required files

Create or update the following files **exactly** as described. Do not skip any file.

### 2.1 `.github/maintainer/.version`

A plain text file containing only the pinned version string (no `v` prefix):

```
0.1.0
```

Replace `0.1.0` with the latest version from Step 1.

---

### 2.2 `.github/maintainer/config.yml`

Fetch the default config template from:

```
https://raw.githubusercontent.com/ianlintner/caretaker/main/dist/templates/config-default.yml
```

Save it to `.github/maintainer/config.yml` and **do not modify it** unless the issue
explicitly requests customization. Remove the comment on line 1 that says "Generated by
caretaker setup — customize as needed." and replace it with a comment identifying this
as the repo-specific config file.

---

### 2.3 `.github/workflows/maintainer.yml`

Fetch the workflow template from:

```
https://raw.githubusercontent.com/ianlintner/caretaker/main/dist/templates/workflows/maintainer.yml
```

Save it verbatim to `.github/workflows/maintainer.yml`. **Do not modify** it.

> **Important:** The template already uses the correct install method
> (`git+https://github.com/ianlintner/caretaker.git`) and passes the event payload
> through an environment variable to avoid bash quoting issues. Do not revert these.

### 2.7 `.github/copilot-instructions.md`

**Append** (do not overwrite) the following block to the end of
`.github/copilot-instructions.md`. If the file does not exist, create it with only this
block:

```markdown
## Caretaker

This repo uses the [caretaker](https://github.com/ianlintner/caretaker) autonomous
maintenance system. The orchestrator runs daily via GitHub Actions and assigns tasks to
`@copilot` via structured issue and PR comments.

Agent instruction files live in `.github/agents/`:
- `maintainer-pr.md` — how to respond to PR fix requests
- `maintainer-issue.md` — how to execute assigned issues
- `maintainer-upgrade.md` — how to apply caretaker upgrades

Always check these files when you receive a caretaker assignment.
```

### 2.8 Fleet registry opt-in (recommended)

Caretaker can roll up runs from every consumer repo into a central admin dashboard at
`https://caretaker.cat-herding.net`. Each consumer opts in by enabling the
`fleet_registry` block in `config.yml` and exporting a shared HMAC secret to the workflow.

**To opt this repo into the fleet:**

1. **Edit `.github/maintainer/config.yml`** — locate the `fleet_registry:` block (already
   present in the template, opt-in disabled by default) and change `enabled: false` to
   `enabled: true`. The default endpoint already points at the production receiver.
2. **Add a repository secret** named `CARETAKER_FLEET_SECRET` in
   *Settings → Secrets and variables → Actions*. Use the same value the caretaker fleet
   admin distributed to other repos (HMAC shared secret). The workflow already reads this
   secret into the `CARETAKER_FLEET_SECRET` environment variable for every job step.
3. **Verify locally** (optional) — after the next caretaker run on the default branch,
   the repo should appear at `/fleet` in the admin dashboard within a minute. You can
   also run `caretaker fleet register-self --config .github/maintainer/config.yml` from a
   checkout to send a one-shot heartbeat for verification.

If the secret is unset, the heartbeat is sent unsigned. The receiver currently accepts
unsigned heartbeats but production deployments should always set the secret.

> **Note for repos that intentionally stay out of the fleet** (e.g. private demos):
> leave `fleet_registry.enabled: false`. No secrets or workflow changes are required.

---

## Step 3 — Open a pull request

After creating all files, open a **single pull request** with:

- **Title:** `chore: setup caretaker`
- **Body:** List all files created/modified

Do not merge the PR yourself. The repo owner will review and merge it.

---

## Notes

- The `ANTHROPIC_API_KEY` secret is optional but unlocks enhanced Claude-backed features
  (CI log analysis, architectural review, issue decomposition). Add it in repo Settings →
  Secrets and variables → Actions if desired.
- The workflow uses `git+https://` to install caretaker directly from GitHub at the
  pinned version tag. This avoids conflicts with the unrelated `caretaker` package on PyPI.
- The event payload is passed through the `CARETAKER_EVENT_PAYLOAD` environment variable
  rather than inline in the shell command. This prevents bash quoting errors when the
  payload contains single quotes (e.g. in PR bodies or commit messages).
